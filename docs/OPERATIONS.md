# Operations guide

Nocturne-KX is a single static binary with no server mode, no configuration file, and no daemon. Every operation is an explicit CLI call; state lives in three kinds of files: key material, the replay database, and the audit log. This guide covers keeping those files healthy.

## File layout

A typical production layout looks like:

```
/etc/nocturne/
  receiver_hybrid_pk.bin      # public (world-readable is fine)
  receiver_hybrid_sk.nchsm2   # secret (0600, owner: nocturne service account)
  auditor_pk.bin              # public (verify audit log)
  auditor_sk.nchsm2           # secret (0600, offline or in HSM)

/var/lib/nocturne/
  replay.db                   # AEAD-encrypted, atomic writes
  replay.db.macsk             # MAC key for replay DB (0600)

/var/log/nocturne/
  audit.log                   # JSONL, append-only
  audit.log.chain             # BLAKE2b chain head (generated by audit-verify)
```

Set permissions immediately after provisioning:

```bash
chown -R nocturne:nocturne /etc/nocturne /var/lib/nocturne /var/log/nocturne
chmod 700 /etc/nocturne /var/lib/nocturne
chmod 600 /etc/nocturne/*_sk* /var/lib/nocturne/replay.db.macsk
chmod 644 /etc/nocturne/*_pk.bin
chmod 755 /var/log/nocturne
chmod 640 /var/log/nocturne/audit.log
```

## Systemd unit

Run nocturne-kx as an unprivileged user. Since the binary is called as a one-shot per message, trigger it from your application layer.

```ini
# /etc/systemd/system/nocturne-decrypt.service
[Unit]
Description=Nocturne-KX decrypt

[Service]
Type=oneshot
User=nocturne
Group=nocturne
EnvironmentFile=/etc/nocturne/env
ExecStart=/usr/local/bin/nocturne-kx decrypt \
  --rx-pk /etc/nocturne/receiver_hybrid_pk.bin \
  --rx-sk /etc/nocturne/receiver_hybrid_sk.nchsm2 \
  --replay-db /var/lib/nocturne/replay.db \
  --audit-log /var/log/nocturne/audit.log \
  --audit-sign-key /etc/nocturne/auditor_sk.nchsm2 \
  --in %i --out %i.plain
NoNewPrivileges=true
PrivateTmp=true
ProtectSystem=strict
ReadWritePaths=/var/lib/nocturne /var/log/nocturne
```

```ini
# /etc/nocturne/env
NOCTURNE_HSM_PASSPHRASE=...   # or use PKCS11_LIB + NOCTURNE_HSM_PIN for real HSM
```

## Key lifecycle

### Initial provisioning

```bash
# On an offline or air-gapped machine
./nocturne-kx gen-receiver /etc/nocturne --kem hybrid
./nocturne-kx gen-signer   /etc/nocturne --sig-type hybrid
```

For production, prefer provisioning keys inside the HSM and never exporting them. See the [HSM guide](./guide/hsm).

### Rotation

There is no background rotation daemon. Rotation is a manual, auditable event:

1. Generate new keys with a new label.
2. Distribute the new public key to all senders.
3. Run `decrypt` with the old key for any in-flight packets.
4. Once all senders have switched, retire the old key and delete the file.

Dual-control rotation via `nocturne::security::KeyRotationManager` (the enterprise API) enforces approval from two operators before a rotation completes. See `src/security/key_rotation.hpp`.

## Replay database

The replay DB is the only mutable state that must survive across restarts. Treat it like a database:

```bash
# Take a point-in-time backup (atomic; rename guarantees DB is consistent)
cp /var/lib/nocturne/replay.db     /backup/replay-$(date +%Y%m%d).db
cp /var/lib/nocturne/replay.db.macsk /backup/replay-$(date +%Y%m%d).macsk
```

::: warning Never share the replay DB across hosts
The DB is single-writer by design. Two processes writing to the same file simultaneously cause AEAD failures and spurious ReplayDetected errors. Use a separate DB per process or serialise through a wrapper.
:::

::: danger Losing the replay DB
If the DB is lost or corrupt, stop processing packets until it is restored from backup. Restarting with an empty DB lets an attacker replay all previously delivered packets. If you have no backup, rotate all receiver keys and reject all pre-rotation ciphertexts.
:::

## Audit log rotation

The audit log is append-only JSONL. Verify the chain before rotating:

```bash
# Verify chain integrity
./nocturne-kx audit-verify /var/log/nocturne/audit.log \
  --expect-signer /etc/nocturne/auditor_pk.bin

# Archive and start fresh
mv /var/log/nocturne/audit.log /var/log/nocturne/audit-$(date +%Y%m%d).log
```

For logrotate:

```
/var/log/nocturne/audit.log {
    daily
    rotate 90
    compress
    missingok
    notifempty
    prerotate
        /usr/local/bin/nocturne-kx audit-verify /var/log/nocturne/audit.log \
          --expect-signer /etc/nocturne/auditor_pk.bin \
          >> /var/log/nocturne/verify.log 2>&1
    endscript
    create 640 nocturne nocturne
}
```

## Monitoring

Nocturne-KX writes no metrics endpoint. Wire it up through exit codes and the audit log:

```bash
# Health check
./nocturne-kx self-test || alert "nocturne self-test failed"

# Audit log SIEM forwarding (syslog UDP/TCP/TLS are wired sinks)
# configure via SIEMConnector in src/security/siem_connector.hpp
```

## Incident response

### ReplayDetected

```
ReplayDetected: counter N <= last seen N for session X
exit 2
```

1. Check the audit log for the original packet's timestamp.
2. If the replay came from an unexpected source, rotate receiver keys and invalidate sessions.
3. If it came from a legitimate sender in a retry loop, investigate why the sender-side counter did not advance.

### AeadAuthFailed

```
AeadAuthFailed: Poly1305 tag mismatch
exit 2
```

Packet was tampered in transit or the wrong receiver key was used. Verify `--rx-pk` and `--rx-sk` match, then treat any mismatch on correct keys as a tamper event.

### SignatureVerifyFailed

```
SignatureVerifyFailed: pinned signer does not match packet
exit 2
```

The packet's signature does not match `--expect-pqc-signer`. Either the sender key rotated without notifying you, or the packet came from an impersonator. Alert and audit.
